# Аудит `app.py` та `nginx.conf`

## Знайдені проблеми та виправлення в `app.py`

### 1. 🔴 За замовчуванням застосунок міг стартувати в debug-режимі
```python
parser.add_argument('--mode', choices=['debug', 'production'], default='debug', ...)
```
Якщо застосунок коли-небудь запускали без явного `--mode production`
(наприклад, вручну після перезавантаження сервера, чи через
Планувальник завдань Windows, який просто викликає `python app.py`) —
він тихо стартував у **Flask debug-режимі**. У цьому режимі Werkzeug
вмикає інтерактивну веб-консоль: при будь-якій необробленій помилці
кожен, хто відкриє сторінку з помилкою, отримує консоль з можливістю
виконати **довільний Python-код прямо на сервері**. Це офіційно
задокументована поведінка Werkzeug, не баг бібліотеки — але оскільки
ваш застосунок дивиться назовні через nginx на 443 порт, це означає
потенційне віддалене виконання коду для будь-кого, хто зловить
помилку на сайті.
**Виправлено**: за замовчуванням тепер `production` (waitress, без
debug-консолі). `debug` потрібно вмикати явно і свідомо — лише на
своїй машині розробника, не на сервері.

### 2. 🟠 `request.remote_addr` завжди показував адресу nginx, а не реального користувача
Застосунок працює за nginx (SSL termination + reverse proxy), який
виставляє заголовки `X-Real-IP` / `X-Forwarded-For`. Але без
`ProxyFix` Flask/Werkzeug їм не довіряють і використовують адресу
безпосереднього з'єднання — тобто **сам nginx** (як правило,
`127.0.0.1`). Це означає, що аудит-логування спроб входу в `auth.py`
(`log_action(..., details=f"IP: {request.remote_addr}")`) для
**кожного** користувача й кожної спроби входу писало один і той же
`127.0.0.1` — журнал ставав марним для розслідування підозрілих спроб
входу.
**Виправлено**: додано `ProxyFix(app.wsgi_app, x_for=1, x_proto=1, x_host=1)`.

### 3. 🟠 Шаблони лежали всередині `static/` - можна було завантажити сирий код сторінки
```python
app = Flask(__name__, template_folder=os.path.join(BASE_DIR, 'static', 'templates'), static_folder=os.path.join(BASE_DIR, 'static'))
```
Flask автоматично роздає **весь** вміст `static_folder` на URL
`/static/<шлях>`. Якщо `templates` лежить усередині `static`, будь-хто
міг звернутися напряму на `https://ваш-сайт/static/templates/login.html`
і отримати сирий, невідрендерений HTML/Jinja-файл сторінки — включно з
коментарями, назвами внутрішніх макросів і структурою логіки, яку не
призначено для публічного перегляду.
**Виправлено**: `template_folder` тепер `<корінь проєкту>/templates`
(поруч зі `static/`, як і задумано в Flask за замовчуванням).
**⚠️ Дія з вашого боку**: фізично перенесіть папку з .html-шаблонами з
`static/templates/` на рівень вище, у `templates/` (поруч зі `static/`),
інакше після оновлення `app.py` Flask не знайде файли шаблонів.

### 4. 🟡 Неточні docstring-и (не відповідали реальним значенням за замовчуванням)
Docstring стверджував `--host` за замовчуванням `192.168.0.219`,
`--port` — `8080`. Реальний код: `default='localhost'`, `default=5000`
— саме ці значення узгоджені з `nginx.conf` (`proxy_pass
http://localhost:5000`). Документація просто застаріла після зміни
коду.
**Виправлено**: docstring приведено у відповідність до коду.

### 5. 🟡 У debug-режимі ігнорувалися `--host`/`--port`
`app.run(debug=True)` викликався без аргументів — якщо хтось передавав
`--host`/`--port` разом з `--mode debug`, вони мовчки ігнорувались.
**Виправлено**: `app.run(debug=True, host=args.host, port=args.port)`.

### 6. 🟡 Необроблені помилки ніде не логувались
Не було жодного `@app.errorhandler` — будь-яка необроблена помилка поза
`try/except` в самих маршрутах просто показувала користувачу 500-у
сторінку, а деталі залишались лише в консолі процесу (яку на
Windows-сервері, запущеному фоново, ніхто не бачить).
**Виправлено**: додано `@app.errorhandler(500)`, який пише повний
traceback у `app.log` через спільний логер, плюс `@app.errorhandler(404)`
з дружньою сторінкою замість стандартної Flask-заглушки. Додано шаблони
`404.html` і `500.html` в єдиному стилі сайту.

### 7. Дрібне
- Прибрано невикористаний імпорт `log_action` (ніде не викликався в `app.py`).
- Додано `SESSION_COOKIE_SECURE=True`, `SESSION_COOKIE_HTTPONLY=True`,
  `SESSION_COOKIE_SAMESITE='Lax'`, `PREFERRED_URL_SCHEME='https'` — сайт
  віддається лише по HTTPS, кука сесії тепер узгоджена з цим.
- Додано `MAX_CONTENT_LENGTH=32MB` (узгоджено з `client_max_body_size`
  у `nginx.conf`, див. нижче) - захист від надто великих запитів.

## Знайдені проблеми та виправлення в `nginx.conf`

### 1. 🔴 Відсутній `client_max_body_size` - імпорт великих файлів обривався ще до Flask
За замовчуванням nginx обмежує тіло запиту **1 МБ**. У проєкті є
кілька сторінок імпорту з Excel (список студентів, предмети, документи
про освіту) — файл з великою кількістю рядків цілком міг перевищити
1 МБ, і тоді nginx повертав `413 Request Entity Too Large` ще ДО того,
як запит взагалі доходив до Flask (тобто це навіть не потрапляло в
`app.log`, виглядало як "сайт просто не відповідає").
**Виправлено**: `client_max_body_size 32m;` (узгоджено з
`MAX_CONTENT_LENGTH` у `app.py`).

### 2. Додано проксі для ONLYOFFICE Document Server
Новий блок `location /onlyoffice/` — проксіює на Document Server,
встановлений на цьому ж сервері (порт 8081, оскільки 80 вже зайнятий
під редірект на HTTPS). Це дозволяє віддавати редактор документів з
того самого домену й сертифіката, що й основний сайт — без проблем із
змішаним контентом чи окремим SSL. Додано заголовки `Upgrade`/
`Connection: upgrade` для WebSocket (потрібно для сповіщень
співавторства в реальному часі) та окремий, більший
`client_max_body_size 50m` саме для цього шляху. Детальніше — у
`ONLYOFFICE_SETUP.md`.

### Що залишили без змін (не критично, але майте на увазі)
- `keepalive_timeout` / `proxy_*_timeout` виставлені у 1400 секунд
  (~23 хвилини) — дуже довго. Ймовірно, навмисно (для довгих операцій
  на кшталт масової генерації документів чи великого імпорту), але
  водночас це потенційний вектор для "повільних" з'єднань, що тримають
  воркер зайнятим довго. Якщо не було конкретної причини для такого
  значення — можна обережно зменшити (наприклад, до 120–300 секунд) і
  подивитися, чи вистачає для найдовших операцій.
- Статичні файли (`/static/...`) віддаються через проксі на Flask,
  а не напряму nginx. Працює коректно, просто трохи менш ефективно, ніж
  могло б бути (nginx набагато швидше роздає статичні файли, ніж
  Python-процес). Не терміново, можна оптимізувати окремим `location`
  пізніше.
