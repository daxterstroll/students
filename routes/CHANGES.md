# Аудит проєкту `students` — знайдені проблеми та внесені правки

Перевірені файли: `admin.py`, `auth.py`, `config.py`, `db.py`, `gen_docx.py`,
`init_db.py`, `students.py`, `update_groups.py`, `utils.py`.
Додано новий файл: **`helpers.py`**.

Нижче — все знайдені невідповідності між файлами (з поясненням, чому це
проблема) і що саме змінено. Всі файли перевірені на коректність
(`ast.parse` + `pyflakes`) — новий код не додає жодних синтаксичних
помилок чи нових невикористаних імпортів.

---

## 1. Реальні баги (проєкт міг падати або працювати неправильно)

### 1.1 `init_db.py` — скрипт ініціалізації бази падав при кожному запуску
```python
for u, p, r, g in [('admin', 'admin123', 'admin', '1', '[]')]:
    cur.execute("INSERT ... VALUES (?, ?, ?, ?, ?)", (u, generate_password_hash(p), r, g))
```
Кортеж мав **5** елементів (логін, пароль, роль, is_admin, permissions), а
розпаковувався лише в **4** змінні `u, p, r, g` → `ValueError: too many
values to unpack`. Навіть якби розпакування спрацювало, в `INSERT`
передавалось 4 значення на 5 плейсхолдерів `(?, ?, ?, ?, ?)` — теж
крах. Тобто **скрипт створення бази даних ніколи не міг успішно
відпрацювати "з коробки"**.
**Виправлено**: акуратний список `DEFAULT_USERS` з 5 полями + `try/except`
навколо `INSERT` з логуванням помилки.

### 1.2 `update_groups.py` ⇄ `init_db.py` — неіснуюча колонка `course`
`update_groups.py` (скрипт автоматичного переведення груп на наступний
курс щороку 1 вересня) читає й оновлює `groups.course`:
```sql
SELECT id, name, course, start_year, program_credits FROM groups WHERE archived = 0
...
UPDATE groups SET course = ? WHERE id = ?
```
Але в схемі `init_db.py` (і ніде більше в коді) колонки **`course` в
таблиці `groups` не було взагалі**. Це означає, що на щойно створеній
базі (або на будь-якій базі, де цю колонку не додавали вручну через
`ALTER TABLE`) скрипт падав щоразу з
`sqlite3.OperationalError: no such column: course`, і **автоматичне
переведення груп на наступний курс могло ніколи фактично не
спрацьовувати** — про це ніхто б не дізнався, бо помилка тихо
записувалась лише в окремий `group_update.log`, який рідко хтось
перевіряє.
**Виправлено**: додано `"course" INTEGER NOT NULL DEFAULT 1` до схеми
`groups` в `init_db.py`.
**⚠️ Дія для існуючої (production) бази даних**: якщо колонки `course` там
ще немає, виконайте одноразово:
```sql
ALTER TABLE groups ADD COLUMN course INTEGER NOT NULL DEFAULT 1;
-- потім вручну виставте правильний поточний курс для кожної активної групи
```

### 1.3 `init_db.py` ⇄ `admin.py` — схема `groups` не відповідала реальному використанню
`admin.py` (`manage_groups`, `generate_group_docs`) читає й пише в
колонки `groups`, яких **не було в `init_db.py`**:
`institution_name_and_status`, `institution_name_and_status_en`,
`entry_requirements`, `entry_requirements_en`, `learning_outcomes`,
`learning_outcomes_en`, `program_includes`, `program_includes_en`.
Це означає, що ваша реальна (production) база даних, скоріш за все, вже
має ці колонки, додані вручну — але **скрипт `init_db.py` для "чистого"
розгортання проєкту на новому сервері їх не створить** і застосунок
одразу впаде на першій же спробі додати/зберегти групу.
**Виправлено**: усі ці колонки додано до `CREATE TABLE groups` в
`init_db.py`, щоб фреш-інсталяція збігалась з тим, що фактично очікує код.

### 1.4 `utils.py` — кеш прав доступу (`permission_required`) фактично не працював
```python
if not hasattr(session, 'is_admin'):
    ...  # підвантажити з БД і закешувати в session
```
`session` у Flask — це словникоподібний об'єкт; звичайного python-атрибута
`is_admin` на ньому **ніколи не існує**, тож `hasattr(session, 'is_admin')`
завжди повертає `False`, а умова `not hasattr(...)` — завжди `True`.
Це означає, що коментар "Спочатку перевіряємо сесію (кеш)" був
оманливим: **кожен захищений `@permission_required(...)` запит робив
зайвий `SELECT` до таблиці `users`**, замість того, щоб брати дані з
сесії, як і задумувалось.
**Виправлено**: `if 'is_admin' not in session:` — тепер кеш дійсно працює.

### 1.5 Небезпечне сортування українською локаллю (2 з 3 місць падали без обробки помилок)
У трьох місцях `admin.py` (`manage_diplomas`, `manage_subjects`,
`manage_activities`) майже однаковий код сортував студентів через
`locale.setlocale(locale.LC_COLLATE, 'uk_UA.UTF-8')`. Лише в одному з
трьох місць був `try/except locale.Error`; у двох інших — ні. Якщо на
сервері не встановлена локаль `uk_UA.UTF-8` (типова ситуація на "чистому"
Linux/Docker-образі) — сторінка падала з 500-ю помилкою.
Так само `db.py`'s `ukrainian_collation()` (яка викликається SQLite під
час КОЖНОГО порівняння рядків в `ORDER BY ... COLLATE UKRAINIAN`)
виконувала `locale.setlocale(...)` на кожен виклик — і теж без обробки
помилок, і вкрай неефективно (сотні/тисячі викликів на один запит).
**Виправлено**:
- Створено `helpers.sort_ukrainian()` — єдина безпечна функція
  сортування з fallback на звичайний порядок, якщо локаль недоступна
  (замість падіння). Усі 3 місця в `admin.py` тепер використовують її.
- `db.py`: локаль встановлюється **один раз** при імпорті модуля (а не на
  кожне порівняння), з тим самим безпечним fallback і попередженням у
  лог, а не винятком.

### 1.6 `log_action()` (`utils.py`) не мала обробки помилок
Якщо `get_db()` чи запит до `groups` падав усередині `log_action()`, виняток
"вилітав" з функції і міг зламати **весь HTTP-запит**, навіть якщо основна
дія користувача (додавання студента, збереження оцінок тощо) вже
завершилась успішно в базі даних. Крім того, з'єднання з БД не завжди
закривалось при помилці (витік з'єднань).
**Виправлено**: `log_action()` тепер обгорнута в `try/except/finally` —
збій запису логу ніколи не ламає основну дію користувача, але сам факт
збою обов'язково записується як `ERROR`.

### 1.7 `auth.py` (`login()`) — з'єднання з БД могло "витікати" при помилці
`conn = get_db()` викликався без `try/finally`; якщо запит до `users`
падав (пошкоджена база, немає прав доступу до файлу тощо), з'єднання
ніколи не закривалось, а користувач бачив "голу" сторінку 500 замість
зрозумілого повідомлення.
**Виправлено**: весь блок роботи з БД у `login()` обгорнуто в
`try/except/finally` з логуванням помилки та дружнім повідомленням.

## 2. Неконсистентність логування між файлами

`admin.py` та `students.py` імпортували **голий** `import logging` і
викликали `logging.error(...)` / `logging.warning(...)`. Але єдиний
файловий логер додатку (з записом у `app.log`) налаштований лише як
`logger = logging.getLogger('Students')` в `utils.py`. Кореневий логер
Python (`logging.error(...)`) обробників не має — такі повідомлення в
кращому випадку виводились у консоль і **ніколи не потрапляли в
app.log**, тобто фактично губились.

**Виправлено**: усі виклики `logging.error/warning(...)` в `admin.py` та
`students.py` замінено на `logger.error/warning(...)` (спільний логер з
`routes.utils`). Тепер усі помилки додатку гарантовано потрапляють в
один `app.log`.

Додатково: у ~12 місцях (`admin.py` та `students.py`) блоки
`except ... as e:` лише показували `flash()` користувачу, але **нічого
не писали в лог** — про такі помилки ніхто, крім самого користувача в
момент збою, ніколи не дізнавався. Додано `logger.error(..., exc_info=True)`
у ці блоки (список: масове призначення періодів навчання, видалення
документа про освіту, редагування документа про освіту, видалення
предмету, `manage_users`, `add_user`, `delete_user`, видалення студента,
збереження оцінок з активностей, імпорт студентів з Excel).

## 3. Дублювання коду — винесено в спільні функції

| Що було продубльовано | Де | Що зроблено |
|---|---|---|
| `session.get('username', 'невідомо')` — 48 разів | `admin.py` (32), `students.py` (17), `auth.py` (1, стилістично) | `helpers.current_username()` |
| Сортування українською локаллю (locale.setlocale + strxfrm) | `admin.py` ×3 | `helpers.sort_ukrainian()` |
| `locale.setlocale` на кожне порівняння в SQL-колації | `db.py` | Локаль встановлюється один раз при імпорті модуля |

Новий файл **`routes/helpers.py`** — єдине місце для цієї спільної
логіки; якщо в майбутньому знадобиться змінити, наприклад, як
визначається ім'я користувача для логів, це робиться в одному місці.

## 4. Безпека / інше

- **`config.py`**: `SECRET_KEY` був захардкоджений прямо в коді
  репозиторію (і, ймовірно, в git-історії). Будь-хто з доступом до коду
  міг підробити сесію будь-якого користувача, включно з адміном.
  Тепер ключ читається зі змінної середовища `SECRET_KEY`; якщо вона не
  задана — генерується тимчасовий ключ (з попередженням у лог), а не
  постійний секрет, видимий у коді. **Для продакшну обов'язково задайте
  свій `SECRET_KEY` в оточенні сервера.**
- **`init_db.py`**: адмін створюється з паролем `admin123` — додано
  явне попередження в консоль одразу змінити пароль після першого входу.

## 5. Що НЕ чіпали навмисно
- HTML-шаблони (`.html`) — вони не були надані, тому не перевірялись.
- Бізнес-логіка запитів (SQL, розрахунки, порядок дій) — залишена
  без змін скрізь, де це не було прямим багом; правки стосуються лише
  обробки помилок, логування та усунення дублювання.
- Дрібні "code smell"-зауваження, що існували ще до правок і не є
  багами (можна виправити пізніше за бажанням):
  - `admin.py`: імпорт `pandas as pd` ніде не використовується.
  - `admin.py`: змінна `is_foreign` в `import_docs_commit` рахується,
    але не використовується.
  - `students.py`: імпорт `transliterate_ukrainian` не потрібен напряму
    (використовується лише всередині `generate_english_name`).
  - `utils.py`: імпорт `re` не використовується.
